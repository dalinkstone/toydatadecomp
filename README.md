# toydatadecomp — Retail Recommendation Engine

A retail recommendation engine that combines real CVS store/product data with
large-scale synthetic data to train a two-tower neural network for purchase prediction.

**Target machine:** M4 Max MacBook Pro, 64GB RAM, 4TB SSD, macOS.

## Data Ecosystem

| Entity         | Source    | Count      | Storage                          |
|----------------|-----------|------------|----------------------------------|
| Stores         | Scraped   | ~9,000     | `data/real/stores.parquet`       |
| Products       | Scraped   | ~12,000    | `data/real/products.parquet`     |
| Customers      | Synthetic | 10M        | `data/synthetic/customers/*.parquet` |
| Coupon Clips   | Synthetic | ~16M       | `data/synthetic/coupon_clips/*.parquet` |
| Transactions   | Synthetic | 10B        | `data/synthetic/transactions/*.csv.zst` |

### Data Relationships

```
stores (real)          products (real)
  ~9K CVS locations      ~12K SKUs (OTC, Rx, grocery, personal care)
       |                      |          |
       v                      v          v
  transactions (synthetic)          coupon_clips (synthetic)
  10B purchase records              ~16M digital coupon clips
  links: customer_id,              links: loyalty_number -> customers
         store_id,                        product_id -> products
         product_id
       |
       v
  customers (synthetic)
  10M profiles with loyalty_number, demographics
```

**Stores** are real CVS locations scraped from cvs.com.
**Products** are a realistic 12K-SKU catalog built from real CVS categories, brands, and pricing.
**Customers** are 10M synthetic profiles (Faker) with loyalty numbers, demographics, and state
distribution proportional to CVS store density.
**Coupon clips** model digital coupon engagement — ~15% of customers clip 3-30 coupons, with
realistic discount types (percent off, dollar off, BOGO) and redemption rates.
**Transactions** are 10B purchase records generated by a multithreaded C binary, linking
customers to products and stores with realistic pricing, quantities, tax, and discounts.

### Known Issue

The C transaction generator assigned sequential store_ids (1, 2, 3...) rather than
real CVS store numbers (e.g. 17435). This means transaction store_ids do not join
to the stores table. All other foreign key relationships are valid. This will need
to be addressed before store-level analytics are meaningful.

## Architecture

```
scrape-stores ───┐
scrape-products ──┤
gen-customers ────┤──> load-db ──> train ──> inference
gen-coupon-clips ─┤       |
gen-transactions ─┘       v
                     DuckDB (cvs_analytics.duckdb)
                       - stores, products, customers, coupon_clips: TABLE
                       - transactions: VIEW over .csv.zst files on disk
                       - 5 analytical views for aggregations
```

### Pipeline Stages

1. **Scrape** — Collect real CVS store locations and product catalog
2. **Generate** — Create 10M customers + 16M coupon clips (Python) and 10B transactions (C)
3. **Load** — Ingest into DuckDB; transactions stay on disk as a VIEW with predicate pushdown
4. **Train** — Train a two-tower neural network on purchase history
5. **Infer** — Score all 10M customers x 10K products for recommendations

### DuckDB Strategy

The `.duckdb` file holds customers, products, stores, and coupon clips (~2.2 GB).
Transactions remain as zstd-compressed CSVs on disk (~200 GB) accessed via a VIEW.
DuckDB reads `.csv.zst` natively with columnar scanning and predicate pushdown,
so the database file stays small while queries over 10B rows still work.

### Disk Optimization

The C transaction generator writes zstd-compressed CSVs via `popen("zstd -", "w")`,
reducing the transaction data footprint from **~1.5TB** (raw CSV) to **~200GB** (.csv.zst).
Parquet conversion is optional (`make convert-parquet`).

## Two-Tower Model

The recommendation model uses a **two-tower architecture**:

- **User tower:** Encodes customer features (demographics, purchase history embeddings)
  into a dense user vector.
- **Item tower:** Encodes product features (category, price, brand) into a dense item vector.
- **Scoring:** Dot product of user and item vectors -> purchase probability.

At inference time, item embeddings are pre-computed and stored in a custom C vector
database (`src/vecdb/`) that uses Apple's Accelerate framework for fast SIMD dot products.
For each of the 10M users, the model retrieves top-K product recommendations via
approximate nearest neighbor search.

## Quick Start

```bash
# Install everything (venv, Python deps, C compilation, zstd check)
make install

# Run the full pipeline end-to-end
make full-pipeline

# Or step by step:
make scrape-stores
make scrape-products
make gen-customers
make gen-transactions
make load-db
make train
make inference

# Optional: convert to Parquet (not required)
make convert-parquet

# Run tests
make test
```

## CLI

```bash
python3 src/cli.py scrape --stores --products
python3 src/cli.py generate --customers --coupon-clips --count 10000000
python3 src/cli.py load
python3 src/cli.py load --materialize    # materialize expensive views as tables
python3 src/cli.py validate
python3 src/cli.py train
python3 src/cli.py infer
```
