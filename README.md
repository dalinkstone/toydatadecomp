# toydatadecomp — Retail Recommendation Engine

A retail recommendation engine that combines real CVS store/product data with
large-scale synthetic data to train a two-tower neural network for purchase prediction.

**Target machine:** M4 Max MacBook Pro, 64GB RAM, 4TB SSD, macOS.

## Data Ecosystem

| Entity       | Source    | Count  | Storage                                   |
|--------------|-----------|--------|--------------------------------------------|
| Stores       | Scraped   | ~9,000 | `data/real/stores.parquet`                 |
| Products     | Built     | ~12,000| `data/real/products.parquet`               |
| Customers    | Synthetic | 10M    | `data/synthetic/customers/*.parquet`       |
| Coupon Clips | Synthetic | ~16M   | `data/synthetic/coupon_clips/*.parquet`    |
| Transactions | Synthetic | 10B    | `data/synthetic/transactions/*.csv.zst`    |

### Data Relationships

```
stores (real)          products (real)
  ~9K CVS locations      ~12K SKUs (OTC, Rx, grocery, personal care)
       |                      |          |
       v                      v          v
  transactions (synthetic)          coupon_clips (synthetic)
  10B purchase records              ~16M digital coupon clips
  links: customer_id,              links: loyalty_number -> customers
         store_id,                        product_id -> products
         product_id
       |
       v
  customers (synthetic)
  10M profiles with loyalty_number, demographics
```

**Stores** are real CVS locations scraped from cvs.com. If scraping fails (CVS blocks
the request), a synthetic fallback generator (`make gen-stores`) creates ~9,000
realistic store locations distributed by the same state weights used for customers.

**Products** are a realistic 12K-SKU catalog built from real CVS categories, brands, and
pricing. The builder runs in `--mode=build` by default (deterministic, always succeeds)
and can optionally attempt `--mode=scrape` against cvs.com.

**Customers** are 10M synthetic profiles (Faker + numpy) with loyalty numbers,
demographics, and state distribution proportional to CVS store density. Generated via
multiprocessing with 8 workers in batches of 100K, output as partitioned Parquet files.

**Coupon clips** model digital coupon engagement — ~10-15% of customers are active
clippers, each clipping 3-30 coupons over a 3-year window (2022-2024), with realistic
discount types (percent off, dollar off, BOGO) and redemption rates.

**Transactions** are 10B purchase records generated by a multithreaded C binary
(`src/generators/txn_generator.c`), compressed on-the-fly with zstd via
`popen("zstd -", "w")`. This reduces the data footprint from ~1.5TB raw CSV to ~200GB
`.csv.zst`.

### Known Issue

The C transaction generator assigned sequential store_ids (1, 2, 3...) rather than
real CVS store numbers (e.g. 17435). This means transaction store_ids do not join
to the stores table. All other foreign key relationships are valid. This will need
to be addressed before store-level analytics are meaningful.

## Architecture

```
scrape-products ──┐
scrape-stores ────┤  (falls back to gen-stores if blocked)
gen-customers ────┤
gen-coupon-clips ─┤──> load-db ──> features ──> train ──> inference
gen-transactions ─┘       |
                          v
                     DuckDB (cvs_analytics.duckdb)
                       - stores, products, customers, coupon_clips: TABLE
                       - transactions: VIEW over .csv.zst files on disk
                       - analytical views for aggregations
```

### Pipeline Stages

1. **Scrape/Build** — Build the product catalog; scrape (or generate) store locations
2. **Generate** — Create 10M customers + 16M coupon clips (Python, multiprocessing) and 10B transactions (C, multithreaded)
3. **Load** — Ingest into DuckDB; transactions stay on disk as a VIEW with predicate pushdown
4. **Features** — Materialize customer and product feature tables for ML
5. **Train** — Train a two-tower neural network on purchase history with margin-weighted loss
6. **Infer** — Score all 10M customers x 12K products for recommendations
7. **Validate** — Run data quality checks (row counts, nulls, referential integrity, value ranges)

### DuckDB Strategy

The `.duckdb` file holds customers, products, stores, and coupon clips (~2.2 GB).
Transactions remain as zstd-compressed CSVs on disk (~200 GB) accessed via a VIEW.
DuckDB reads `.csv.zst` natively with columnar scanning and predicate pushdown,
so the database file stays small while queries over 10B rows still work.

Parquet conversion is optional (`make convert-parquet`) — DuckDB reads `.csv.zst`
natively, so it's not included in the default pipeline. Use `make full-pipeline-parquet`
if you want faster repeat queries; it converts and then deletes the `.csv.zst` files.

## Two-Tower Model

The recommendation model uses a **two-tower architecture** (`src/ml/two_tower.py`):

**Customer tower** (89-dim input → 256-dim L2-normalized embedding):
- `customer_id` → Embedding(10M+1, 64)
- `age` → normalized float
- `gender` → one-hot (M, F, NB)
- `state` → Embedding(51, 16)
- `is_student` → binary
- `total_spend`, `coupon_engagement_score`, `coupon_redemption_rate`, `avg_basket_size` → floats

**Product tower** (103-dim input → 256-dim L2-normalized embedding):
- `product_id` → Embedding(12K+1, 64)
- `category` → Embedding(27, 16)
- `brand` → Embedding(321, 16)
- `price`, `is_store_brand`, `popularity_score`, `margin_pct`, `coupon_clip_rate`, `coupon_redemption_rate`, `organic_purchase_ratio` → floats

**Scoring:** Dot product of L2-normalized customer and product embeddings → affinity score.
**Loss:** BCE with logits, optionally weighted by product margin for revenue optimization.
Negative sampling: 4 random products per positive pair.

At inference time, embeddings are pre-computed and stored in a custom C vector database
(`src/vecdb/`) that uses Apple's Accelerate framework for fast SIMD dot products.

Three inference modes:
- `full-matrix` — Chunked (10M × 256) @ (256 × 12K) matrix multiply
- `per-product` — VecDB top-K customers for each product
- `per-customer` — VecDB top-K products for each customer

## Quick Start

```bash
# Install everything (venv, Python deps, C compilation, zstd check)
make install

# Run the full pipeline end-to-end
make full-pipeline

# Or run a demo at 1/1000th scale (~2 min)
make demo
```

### Step by Step

```bash
make install
make scrape-products         # Build 12K product catalog (always works)
make scrape-stores           # Scrape CVS locations (or: make gen-stores)
make gen-customers           # Generate 10M synthetic customers
make gen-transactions        # Generate 10B transactions (.csv.zst, ~200GB)
make load-db                 # Load into DuckDB (reads .csv.zst directly)
make train                   # Train two-tower model
make inference               # Score 10M × 12K recommendations
make validate                # Run data quality checks
make status                  # Show pipeline status
make test                    # Run pytest suite
```

### Optional

```bash
make convert-parquet         # Convert .csv.zst → Parquet (faster repeat queries)
make full-pipeline-parquet   # Full pipeline with Parquet conversion
```

## CLI

The project provides a Click CLI (`src/cli.py`) with subcommand groups:

```bash
# Scrape/build data
python3 src/cli.py scrape stores
python3 src/cli.py scrape products --mode build

# Generate synthetic data
python3 src/cli.py generate customers --count 10000000
python3 src/cli.py generate stores              # fallback if scraping fails
python3 src/cli.py generate transactions

# Convert formats
python3 src/cli.py convert parquet

# Database
python3 src/cli.py load
python3 src/cli.py load --materialize           # materialize expensive views

# ML
python3 src/cli.py train --epochs 5 --device auto
python3 src/cli.py infer --mode full-matrix --top-k 100

# Inspection
python3 src/cli.py validate
python3 src/cli.py status
```

The `status` command works at any point in the pipeline and shows a rich table of
what's been completed and what's pending:

```
┃ Component    ┃ Status ┃         Rows ┃     Size ┃ Path
```

## Compilation

The C components compile on macOS with:

```bash
# Transaction generator
clang -O3 -march=native -o txn_generator txn_generator.c -lm -lpthread

# Vector database
clang -O3 -march=native -shared -o vecdb.dylib vecdb.c -framework Accelerate
clang -O3 -march=native -o test_vecdb test_vecdb.c vecdb.c -framework Accelerate -lm
```

`make compile-c` handles this automatically.

## Testing

```bash
make test    # runs: pytest tests/ -v
```

The test suite (32 tests) covers:

- **test_generators.py** — Customer schema, distributions, uniqueness (1000 customers);
  product catalog count, categories, price ranges, brand coverage (12K products)
- **test_vecdb.py** — Insert + query correctness, top-K sorting, save/load roundtrip,
  10K-vector benchmark
- **test_pipeline.py** — End-to-end mini pipeline: generate 1000 customers + 100 products
  + 100K transactions, load into DuckDB, verify joins and aggregations, train two-tower
  for 1 epoch, run inference, verify output shapes. Completes in ~2 seconds.
